# 代码 Agent 原型开发方案（修订版 v2）

> 本版相对初版的主要变更（先看这一节，理解为什么大改）：
>
> 1. **核心架构从「固定流水线」改为「带工具的 Agent 推理循环」**。初版把流程写成 `扫描→定位→计划→Patch→测试→修复` 的线性 DAG，但真正能干活的 Code Agent（Claude Code、Aider、SWE-agent）都是 ReAct 式循环：给 LLM 一组工具，由它自己决定每一步。这是本版最根本的改动，后面很多设计都由它派生。
> 2. **MVP（前 4 周）砍掉 Go Executor 和向量 RAG**。单用户单仓库原型几乎没有并发需求，瓶颈是 LLM 网络延迟而非 CPU；安全边界也不在语言而在是否真做了隔离。Python `subprocess` 完全够用。Go/沙箱后移到产品化阶段。
> 3. **修掉初版三处时间表自相矛盾**（Go Executor 到底在 MVP2 还是 MVP3、4 周覆盖到哪个阶段、自动修复与 Go 的先后），统一为一张时间表。
> 4. **新增初版完全空缺、但决定项目成败的工程基础设施章节**：代码编辑策略、评估闭环（eval harness）、可观测性/trace、上下文与成本工程、预算与失败兜底、状态持久化与回滚、模型/API 决策清单。
> 5. **纠正若干技术误判**：「命令白名单 = 安全」不成立；「测试通过 = 改对了」不成立；检索不应只押向量 RAG；上下文应按 token 预算而非文件个数。

---

## 1. 背景与目标

当前已经具备：

- Python Agent 原型
- RAG 检索能力
- LLM 调用能力
- Tool Calling 基础能力
- 多 Agent 系统的初步理解

目标：开发一个 **代码 Agent 原型**，能自动分析代码仓库、定位相关代码、生成并应用代码修改、运行测试，并根据测试结果自动修复一到多轮。

落地优先级：**先让单 Agent 闭环在真实仓库上"真的能改对几个任务"，再谈多 Agent 与产品化**。原型阶段的成功标准是「在固定评测集上的解决率」，不是「功能清单打勾」。

---

## 2. 一个核心认知：Code Agent 是「工具循环」，不是「固定流水线」

这是整份方案的地基，先讲清楚。

初版的线性管道在真实任务上会立刻撞墙：
- "我得再读一个文件才能决定改哪" —— 管道没有回头能力。
- "计划错了要重新规划" —— 管道只能一条道走到黑。
- "探索和修改本应交错" —— 管道把它们硬切成两段。

正确的形态是 **ReAct 式 agent loop**：

```text
系统提示 + 任务需求 + 仓库概览
        ↓
┌──────────────────────────────────────┐
│  LLM 决定下一步动作（调用某个工具）        │
│     ↓                                  │
│  执行工具（read_file / grep / edit /    │
│           run_test / ...）             │
│     ↓                                  │
│  把工具结果喂回对话历史                    │
│     ↓                                  │
│  LLM 继续决定……                         │
└──────────────────────────────────────┘
        ↓（LLM 主动 finish，或触发预算上限）
输出最终 Diff + 执行日志 + trace
```

「规划」「定位」「修改」「测试」「修复」在这个模型里不再是写死的阶段，而是 LLM **可以按需、反复、交错调用的工具**。多 Agent（后期）只是把这个循环里的角色拆开，不是另起炉灶。你现有的 Tool Calling 能力正好就是干这个的。

## 3. 技术选型与语言分工

分工本身合理，但要**明确每种语言进入的阶段**，避免在 MVP 过早引入跨语言复杂度。

| 语言 | 职责 | 进入阶段 |
|---|---|---|
| **Python** | Agent 大脑：需求理解、仓库分析、检索、Prompt、代码生成、编辑、错误分析、修复策略，以及 MVP 阶段的命令执行 | **MVP（第 1 天）** |
| **Go** | 执行/隔离底座：沙箱、进程隔离、单二进制部署、资源限制 | **产品化阶段** |
| **TypeScript** | IDE 交互层：VS Code 插件、Diff 展示、人工确认 UI | **产品化阶段** |

关于 Go 的定位修正（重要）：
- Go 的真正价值是**进程隔离 / 单二进制 / 安全边界**，不是初版写的"并发能力强"——单任务场景并发是伪需求。
- "用 Go 防止 Python 执行危险命令"逻辑不成立：Python 调 Go、Go 再执行，Python 照样决定让 Go 跑什么；**安全边界取决于 Go 是否真做了沙箱，而不是换了语言**。
- 因此 MVP 不引入 Go：Python `subprocess` 同样能做超时、捕获 stdout/stderr、限制工作目录。等真要做容器/namespace 级隔离时再上 Go，那时它才物有所值。

## 4. 阶段划分与统一时间表（修掉初版矛盾）

初版三处时间表打架，本版统一为下面这一张，**作为唯一事实源**：

| 阶段 | 内容 | 累计周期 | 备注 |
|---|---|---|---|
| **阶段一 · 单 Agent 闭环 MVP** | Agent loop + 工具集 + 编辑 + 测试 + 自动修复 + eval + trace（**纯 Python**） | **0–4 周** | 本方案的重点 |
| 阶段二 · 检索与稳健性增强 | repo map 检索、全量回归、best-of-N、上下文压缩调优 | 4–7 周 | 仍纯 Python |
| 阶段三 · 多 Agent 协同 | Planner / Repo / Coder / Tester / Reviewer / Fixer 拆分 | 7–10 周 | 先 Python 编排 |
| 阶段四 · 产品化 | Go 沙箱/Executor、VS Code 插件、流式、权限 | 2–3 个月 | 此时才引入 Go / TS |

明确边界，避免误读：**「4 周目标」只覆盖阶段一，不含多 Agent，也不含 Go。**

## 5. MVP 范围（4 周）

### 做什么

```text
1. Agent loop 骨架（ReAct，工具调用驱动）
2. 一组工具：read_file / list_dir / grep / edit / run_command / finish
3. 可靠的代码编辑（编辑格式 + apply 重试 + 改后校验）—— 见 §6
4. 在临时 git worktree/副本里操作 + 改前建 commit 检查点 + 可回滚 —— 见 §10、§13
5. 测试基线 + 运行测试 + 解析结果 + 自动修复循环 —— 见 §11
6. 多维预算与失败兜底（步数/token/时间、循环检测、优雅放弃）—— 见 §12
7. 评估闭环 eval harness（3–10 个固定任务，量化解决率）—— 见 §14
8. 结构化 trace（每次 LLM 调用的输入/输出/token/耗时/成本）—— 见 §15
9. Prompt caching（成本/延迟的最大杠杆）—— 见 §13
```

### 不做什么（明确推迟）

```text
Go Executor / 完整沙箱        → 阶段四
向量 RAG 作为主力检索          → MVP 用 grep + repo map 就够，向量为可选补充
多 Agent 图编排               → 阶段三
VS Code 插件 / 复杂 UI        → 阶段四
权限系统 / 多人协作 / 云部署     → 阶段四
```

砍掉 Go 和向量 RAG 省下的时间，全部投到 **agent loop、编辑可靠性、eval、上下文/成本工程**——这四块才决定 Code Agent 能不能真干活。

## 6. 代码编辑策略（MVP 第一优先级，初版几乎没碰）

让 LLM 直接产 unified diff 极易失败（行号错位、上下文不匹配、`git apply` 报错），这是 Code Agent 最经典的工程难题。

**编辑格式选型**（建议二选一，先做 SEARCH/REPLACE）：

- **SEARCH/REPLACE 块**（推荐，Aider 同款）：LLM 输出"在文件 X 中，把这段原文替换成那段"，由代码做精确/模糊匹配后替换。比 diff 鲁棒得多。
- **整文件重写**：小文件直接让 LLM 输出新全文。简单可靠，但费 token，且大文件不适用。

**apply 必须有重试与校验**：
- 匹配失败 → 把"没找到这段原文"作为错误反馈给 LLM，让它重出，而不是直接崩。
- 改完做**语法/类型校验**（Python 用 `ast.parse` / `py_compile`，JS/TS 用 `tsc --noEmit` 等）；语法都不过就别浪费一次测试运行，直接回退让 LLM 修。

## 7. Agent 循环与工具集设计

MVP 工具集（够用即可，别一次堆太多）：

| 工具 | 作用 | 关键点 |
|---|---|---|
| `list_dir(path)` | 看目录结构 | 过滤 `.gitignore` / `node_modules` / 二进制 |
| `read_file(path, start, end)` | 读文件片段 | **按行区间读**，不要整文件灌；带行号返回 |
| `grep(pattern, glob)` | 搜代码 | 底层用 ripgrep；这是 MVP 主力定位手段 |
| `edit(path, search, replace)` | 改代码 | 走 §6 的编辑格式 + 校验 |
| `run_command(cmd)` | 跑测试/构建 | 走 §10 的执行与超时 |
| `finish(summary)` | 主动结束 | 触发输出最终 Diff |

工具结果回灌的格式要稳定、可截断（大输出先裁剪再喂回，见 §11）。

## 8. 代码检索策略（别只押向量 RAG）

代码语义检索 ≠ 文档 RAG。实践经验：

- **主力：repo map + grep**。用 tree-sitter 生成一张「全仓符号地图」（有哪些类/函数、在哪个文件），先给 LLM 全局观，再让它按需 `grep`/读片段。这是 Aider 等项目验证有效的路子。
- **可选补充：向量检索**。仅在"按语义找相似实现"时有用，不作为 MVP 主力。
- **大仓库处理**：不可能全量 embedding/全扫；靠符号地图 + 按需检索控制规模。
- 统一过滤 `.gitignore`、`node_modules`、`vendor`、二进制、超大文件。

## 9. 执行与安全（MVP 的务实姿态）

不做完整沙箱，但要有**最低限度隔离**：

```text
1. 只在临时 git worktree 或仓库副本里操作，绝不直接动用户主工作区
2. 改动前 git commit / stash 建检查点，任何时候可回滚
3. 默认断网执行测试命令（防止 LLM 触发的命令外联）
4. subprocess 设超时、限制工作目录、捕获 stdout/stderr/exit code
```

**纠正「白名单 = 安全」的误区**：允许 `pytest` 就等于允许任意代码执行（`conftest.py`）；`git -c core.pager=…`、hooks 可执行任意命令；`npm test` 跑 `package.json` 里任意脚本。所以命令名白名单**不是安全机制**，真正隔离要靠容器/namespace（阶段四 + Go）。MVP 的安全感来自"在一次性 worktree + 断网 + 可回滚"，而不是字符串匹配。

Windows 注意：白名单/路径/shell 差异（cmd vs PowerShell vs bash）、`git apply` 的 CRLF 行尾问题都要适配——你本机是 Windows，worktree 方案天然契合（可复用你现有的 worktree 工作流）。

## 10. 自动测试与修复闭环（补齐初版空白）

```text
            ┌──────── 改前先跑一次「测试基线」 ────────┐
            │  记录原本就红/没有测试的用例              │
            ↓                                       │
生成编辑 → 应用 → 语法校验 → 运行相关测试 → 解析结果      │
            ↓                                       │
   与基线对比：新增几个绿？有没有把原来绿的弄红？          │
            ├── 通过且无回归 → 输出最终 Diff           │
            └── 失败 → 裁剪测试输出 → 喂回 LLM → 再修 ──┘
```

初版漏掉、本版补上的关键点：

- **测试基线**：改之前先跑一遍。否则无法区分"我改坏的"和"仓库原本就坏的"。
- **结果判定**：优先用 exit code，必要时解析框架输出（pytest/go test/jest 格式各不同）。
- **回归检测**：跑相关（或全量）测试，防止"修好 A 改坏 B"，不能只跑目标用例。
- **输出裁剪**：测试日志动辄上万行，直接塞会爆上下文；要先摘要/截取关键报错再喂回。
- **多轮修复的状态**：明确每轮是基于上一轮改动继续，还是回退重来；并保留"修复让结果更糟就回退到上一个最好状态"的能力（依赖 §13 检查点）。

## 11. 上下文与成本工程（初版零提及，却是命门）

- **Prompt caching**：系统提示 + 仓库上下文在多轮高度重复，开缓存可大幅削减成本与延迟，**MVP 第一版就用**。
- **上下文压缩**：agent 跑十几步后历史会撑爆窗口，需要 compaction（摘要旧历史、不重复读同一文件）。
- **token 预算而非文件个数**：初版"5–15 个文件"改为"总 token 预算 + 按行区间按需读取"。
- **成本可见**：每次运行统计 LLM 调用次数、token、花费，进 trace（见 §15）。多 Agent 阶段调用量翻倍，必须有成本意识。

---

## 12. 预算、兜底与失败处理

只限"修复轮数"远远不够。需要多维度：

```text
预算（任一超限即停）：
  max_steps         例如 30–50 步
  max_tokens        总 token 上限
  max_wallclock     例如 5–10 分钟
  max_fix_rounds    例如 3（仅修复子循环）

兜底：
  循环检测     反复改同一行 / 反复跑同一个挂的测试 → 识别并跳出
  优雅放弃     到上限时输出"已完成到哪、还差什么"，不硬撑、不崩溃
  幻觉处理     LLM 编造的文件/函数 → 工具层返回明确错误让它纠正
  越界保护     编辑触及任务范围外的文件 → 拦截或要求确认
```

---

## 13. 状态持久化与回滚

- 一次任务可能几分钟、十几次 LLM 调用，中途崩溃要能 **resume**：checkpoint 已读文件、已做修改、对话历史。
- 回滚不是阶段四的事，**MVP 就要有**：基于 §9 的 git 检查点，随时回到任意中间态或初始态。
- 人工确认开关：MVP 提供 `--dry-run` / "应用前给我看 diff"，既是信任来源也是误删安全阀。

---

## 14. 评估闭环 eval harness（落地与 demo 的分水岭）

**没有它你就是在盲调 prompt。** 这是初版最该有却没有的东西。

```text
1. 固定 3–10 个真实任务，每个带"标准答案"：
   - golden patch，或"这组测试应从红变绿"
2. 一键跑全套，自动打分：解决率 / 测试通过率
3. 每次改 prompt / 换模型 / 调编辑格式，都重跑，看分数涨跌
4. 作为回归基线，防止"改 A 退 B"
```

建议第 1–2 周就搭起来，哪怕只有 3 个任务。后期可对接 SWE-bench / SWE-bench-lite 量化解决率。

**验收标准从"能做某动作"升级为量化指标**：原型合格线 = 在自建评测集上稳定解决 ≥ N/总数，而非功能打勾。

## 15. 可观测性与 trace

初版只有"保存日志"，不够。需要**每一步 LLM 调用的结构化 trace**：输入 prompt、输出、调用的工具、token、耗时、成本。一次任务十几次调用，出问题时要能逐步回放"它当时在想什么"。

落地：接现成方案（LangSmith / Langfuse）或自写 JSONL trace + 简单查看器即可。

## 17. 多 Agent 阶段（阶段三，后期）

单 Agent 闭环在 eval 上稳定后，再把循环里的角色拆开：

```text
Planner   分析需求、制定修改计划
Repo      分析仓库、定位相关文件
Coder     生成代码修改
Tester    运行测试、解析报错
Reviewer  检查质量与风险
Fixer     根据失败继续修复
```

先用 Python 串联编排，不急着上 Go Orchestrator。引入并发的真正价值场景是 **best-of-N**：同一任务并行跑 N 个独立尝试，选测试通过/质量最好的——用 `asyncio` 即可，与 Go 无关。

---

## 18. 产品化阶段（阶段四，引入 Go / TS）

此时才有必要引入 Go 与 TypeScript：

```text
Go：    容器/namespace 级沙箱、Executor、资源限制、单二进制部署
TS：    VS Code 插件、Chat UI、Diff 展示、一键应用、人工确认、流式状态
其他：   权限控制、回滚、多项目、云端部署
```

Go Executor 对外可暴露 HTTP API 或本地 CLI（先 CLI 简单，要工程化用 HTTP）。但再次强调：**它的价值是真隔离，不是"换语言换来的安全"**。

## 20. 修订后的 4 周时间表（每周末有可量化验收）

| 周 | 重点 | 周末验收 |
|---|---|---|
| **第 1 周** | Agent loop 骨架 + 工具集 + **eval harness（3 个任务）** + trace；模型/API 决策拍板 | loop 能在 1 个简单任务上跑通"读→改→输出 diff"，eval 能出分 |

## 21. 总结

修订后的落地主线：

```text
先把「单 Agent 工具循环」在真实仓库上做到 eval 解决率达标（纯 Python，4 周）
  → 再做检索与稳健性增强（repo map / 回归 / best-of-N）
    → 再拆多 Agent（Python 编排）
      → 最后产品化时才引入 Go 沙箱与 TS 插件
```

技术组合最终仍是 `Python + Go + TypeScript`，但**进入时机被明确分层**：

```text
Python：Agent 大脑（MVP 第 1 天）
Go：    真隔离与执行底座（产品化阶段，价值在隔离而非并发/换语言）
TS：    IDE 交互层（产品化阶段）
```

最有价值的近期目标（修订后）：

> **4 周内做出一个纯 Python 的单 Agent 工具循环：能在真实仓库上分析、定位、可靠编辑、跑测试、自动修复，并在一个固定评测集上拿到可量化的解决率。**
>
> 比起"功能清单打勾"，「eval 上能解决几个真实任务」才是这个原型真正的辨识度，也是后续一切优化的标尺。
